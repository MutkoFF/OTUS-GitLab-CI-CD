## Flow разработки

### Цели:
1) Напишите, какие ошибки вы видите в процессе разработки. Какой совет (ы) вы можете дать разработчику;

2) Опишите два примера разработки с разными вариантами flow.


**Ответ на первый вопрос**

Во-первых, разработчик нарушает одно из главных правил микросервисов и собирает все приложения в один контейнер. Это противоречит лучшим практикам контейнеризации и принципам Twelve-Factor App, поэтому могут возникнуть следующие проблемы:

1) Невозможно масштабировать отдельные части системы (например, только бэкенд или только обработку заказов);
2) При падении одного компонента падает всё приложение;
3) Усложняется обновление и сопровождение;
4) Один контейнер не получится легко масштабировать при росте нагрузки, что со временем приведет к деградации производительности;
5) Так как скорее всего, состояние приложения хранится внутри изолированной среды, контейнеры будет сложно перезапускать и масштабирование становится невозможным без потери данных.

В целях повышения производительности работы приложения, я могу дать следующие рекомендации:

1) Разделить приложение на несколько контейнеров, например: API бэкенда, БД, брокер сообщений, сервис авторизации и т.д.
2) Следовать принципам Twelve-Factor App: Хранить конфигурацию в переменных окружения, использовать stateless-процессы, рассматривать сторонние сервисы как ресурсы;
3) Для горизонтального масштабирования, управлением нагрузкой и отказоустойчивости использовать оркестратор контейнеров.

**Ответ на второй вопрос**

**Пример 1: Agile + Git Flow в сценарии, когда разработка интернет-магазина с постоянными изменениями требований от заказчика.**

Flow работы:

1) Ветка main — стабильная версия продукта;

2) Ветка develop — активная разработка;

3) Для каждой новой функции создаётся ветка feature/*;

4) Разработка ведётся итерациями (спринтами);

5) После завершения спринта:

   5.1) Feature-ветки сливаются с develop;

   5.2) Создаётся release-ветка.

После тестирования релиз перетекает в main;

Срочные исправления делаются через hotfix.

Данный подход подойдёт потому что:

1) Требования часто меняются;
2) Нужна быстрая доставка функциональности;
3) Есть чёткий контроль релизов.

**Пример 2: Waterfall + V-модель в сценарии разработки платёжного сервиса с высокими требованиями к качеству и безопасности.**

Flow работы:

1) Полный сбор и фиксация требований;

2) Проектирование архитектуры;

3) Реализация;

4) Параллельное тестирование:

   4.1) Модульное тестирование;

   4.2) Интеграционное тестирование;

   4.3) Системное тестирование.

5) Внедрение;

6) Сопровождение.

Данный подход подойдёт потому что:
1) Требования заранее известны;
2) Изменения минимальны;
3) Критически важно качество и надёжность;
4) Строгий контроль тестирования на каждом этапе.
